# PETSc Benchmarking Scripts

This directory contains build and job submission scripts for benchmarking the PETSc OpenMP backend on ARCHER2 and related systems.

## Overview

These scripts are part of an MSc dissertation project investigating the performance characteristics of the PETSc OpenMP backend compared to MPI-only execution on many-core systems. The benchmark application is based on `PETSc/src/ksp/ksp/tutorials/ex2.c`, which solves a 2D five-point finite difference problem.

## File Structure

### Environment Setup

- **`env_archer2_gnu.sh`** ŌĆö Verifies and displays the current compiler and MPI environment on ARCHER2
  - Loads required modules (PrgEnv-gnu, cray-mpich)
  - Displays compiler versions and paths
  - Checks OpenMP and Slurm environment variables

- **`config.txt`** ŌĆö PETSc configuration command for building with OpenMP support
  - Configures PETSc 3.24.4 with OpenMP and OpenMP kernels enabled
  - Uses `-O3` optimization flags
  - Includes FBLASLAPACK support

- **`ENVIRONMENT.md`** ŌĆö Detailed documentation of the software environment and PETSc configuration
  - ARCHER2 platform specifications
  - Module and compiler details
  - PETSc build options
  - Benchmark setup and execution modes

### Slurm Job Submission Scripts

All scripts are designed for submission to ARCHER2 using `sbatch`:

#### Pure Execution Modes

- **`run_ex2_mpi.sbatch`** ŌĆö Pure MPI execution (1 rank, 4 tasks, 1 core per task)
  - Tests baseline MPI-only performance
  - Can be adapted for different rank counts

- **`run_ex2_omp.sbatch`** ŌĆö Pure OpenMP execution (single rank, 128 threads)
  - Tests baseline OpenMP-only performance
  - Useful for thread-scaling analysis

- **`run_ex2_mpi_min.sbatch`** ŌĆö Minimal MPI configuration
  - Basic setup for quick validation

#### Hybrid MPI+OpenMP Configurations

- **`run_ex2_grid.sbatch`** ŌĆö Sweeps through a grid of rank-thread combinations
  - Tests multiple configurations in a single job
  - Evaluates different hybrid decompositions
  - Generates comparative performance data
  - Uses the original fixed validation size unless edited directly

- **`ex2_problem_sizes.csv`** ŌĆö Problem-size table for larger 2D five-point stencil tests
  - Defines named scales from 1.00M to 40.96M unknowns
  - Used by `run_ex2_size_grid.sbatch`

- **`run_ex2_size_grid.sbatch`** ŌĆö Sweeps selected problem sizes across rank-thread combinations
  - Reads `ex2_problem_sizes.csv`
  - Writes one CSV with `scale`, `m`, `n`, and `unknowns` columns
  - Stores logs under `runs/size_grid/<scale>/`
  - Defaults to `SCALES=small` to avoid accidental large submissions

- **`run_ex2_fixed128_nodes1.sbatch`** ŌĆö Single-node, fixed to 128 cores (all cores on one node)
  - Tests different rank-thread distributions on a single node
  - Problem size: 800 ├Ś 800 (640,000 unknowns)

- **`run_ex2_fixed128_nodes2.sbatch`** ŌĆö Multi-node configuration with 2 nodes
  - Fixed 128 cores total, distributed across 2 nodes
  - Tests impacts of NUMA and inter-node communication

- **`run_ex2_fixed128_nodes4.sbatch`** ŌĆö Multi-node configuration with 4 nodes
  - Fixed 128 cores total, distributed across 4 nodes
  - Tests scalability with increased node count

## Usage

### Prerequisites

1. **Load the environment**:
   ```bash
   source env_archer2_gnu.sh
   ```

2. **Ensure PETSc is built** with OpenMP support:
   ```bash
   cd $PETSC_DIR
   ./configure $(cat config.txt | head -1)
   make -j 8 all
   ```

3. **Set paths** in job scripts as needed:
   - `PETSC_DIR` ŌĆö Path to PETSc installation
   - `PETSC_ARCH` ŌĆö PETSc architecture directory

### Submitting Jobs

Submit a job to ARCHER2:

```bash
sbatch run_ex2_grid.sbatch
```

Submit the larger problem-size sweep:

```bash
# Conservative first pass: 1.00M unknowns only
sbatch run_ex2_size_grid.sbatch

# Run one named scale
SCALES="medium" sbatch run_ex2_size_grid.sbatch

# Run the larger scales after the smaller cases look healthy
SCALES="large very-large" sbatch run_ex2_size_grid.sbatch

# Full table, repeated timings
SCALES="small medium-small medium large very-large" REPEAT=3 sbatch run_ex2_size_grid.sbatch
```

For very large cases, consider narrowing the rank/thread grid first:

```bash
SCALES="very-large" RANKS="16 32 64 128" THREADS="1 2 4 8" sbatch run_ex2_size_grid.sbatch
```

Monitor job status:

```bash
squeue -u $USER
```

View output and errors (generated by `#SBATCH --output` and `#SBATCH --error` directives):

```bash
cat petsc_ex2_grid.JOBID.out
cat petsc_ex2_grid.JOBID.err
```

## Key Environment Variables

The scripts set or control the following OpenMP environment variables:

- **`OMP_NUM_THREADS`** ŌĆö Number of OpenMP threads per rank
- **`OMP_PLACES`** ŌĆö Resource placement hint (cores)
- **`OMP_PROC_BIND`** ŌĆö Thread binding policy (close)

These are typically determined dynamically from Slurm settings (`SLURM_CPUS_PER_TASK`).

## Performance Tuning Tips

1. **Thread binding**: Ensure `OMP_PROC_BIND=close` for NUMA-aware systems like ARCHER2
2. **Affinity**: Use `OMP_PLACES=cores` to bind threads to specific cores
3. **Problem size**: Current default is 800 ├Ś 800; adjust in job scripts for larger studies
4. **Timing**: Long-running jobs may timeout; adjust `--time` in Slurm scripts as needed
5. **Exclusive execution**: Uncomment `##SBATCH --exclusive` in scripts to reserve full nodes (if policy allows)

## Larger Problem Sizes

The current large-size study uses the following 2D five-point stencil scales:

| Scale | Grid size | Unknowns |
| --- | ---: | ---: |
| small | 1000 x 1000 | 1.00M |
| medium-small | 2240 x 2240 | 5.02M |
| medium | 3160 x 3160 | 9.99M |
| large | 4500 x 4500 | 20.25M |
| very-large | 6400 x 6400 | 40.96M |

Use `run_ex2_size_grid.sbatch` for these cases rather than editing `M` and `N` repeatedly in `run_ex2_grid.sbatch`. This keeps the experimental design reproducible and keeps the scale metadata in the CSV output.

Recommended progression:

1. Run `small` with the full rank/thread grid as a smoke test.
2. Run `medium-small` and `medium` with `REPEAT=1` to check runtime and memory.
3. Run `large` and `very-large` with a narrower grid first, for example `RANKS="16 32 64 128"` and `THREADS="1 2 4 8"`.
4. Increase `REPEAT` only after the selected configurations fit comfortably within the wall time.

## Troubleshooting

- **PETSc not found**: Check `PETSC_DIR` and `PETSC_ARCH` paths in scripts
- **Module load errors**: Run `module avail` to check available compiler modules
- **Job timeout**: Increase the `--time` parameter in the Slurm header
- **Poor scaling**: Check OpenMP affinity settings and node load balance
- **Memory issues**: Verify problem size and number of processes/threads
